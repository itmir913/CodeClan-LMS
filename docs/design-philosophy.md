# CodeClan LMS — 디자인 철학

## Vision

> **"코드에만 집중하게 하라"**
>
> 교사는 3번의 클릭으로 수업을 시작하고, 학생은 화면 밖으로 눈을 돌릴 필요가 없다.
> 구글 클래스룸이 해결하지 못한 정보 교과의 불편함을 제거한다.
> 지금은 심플하게 시작하고, 구글 클래스룸을 완전히 대체하는 것이 장기 목표다.

---

## 4대 설계 원칙 (Design Pillars)

### Pillar 1 — Contextual Unity (맥락 통합)
학생은 LMS·IDE·컴파일러를 오가지 않는다. 문제 지문 + 코드 에디터 + 실행 결과가 한 화면에 존재한다. 교사는 수업을 선택하면 모든 도구가 그 컨텍스트 안에 있다.

### Pillar 2 — Effortless Authority (손쉬운 통제)
교사는 `수업 선택 → 세션 탭 → 시작 버튼` 3번으로 수행평가를 시작한다. 복잡한 설정은 템플릿과 기본값으로 숨긴다.

### Pillar 3 — Instant Truth (즉각적 진실)
채점 결과는 텍스트보다 색·아이콘이 먼저다. 단, 색만으로 상태를 전달하지 않는다 — 배지(badge)에 색 + 텍스트(Correct / Wrong / Error)를 항상 함께 표시한다. 색각 이상 사용자도 동일한 정보를 얻어야 한다.

### Pillar 4 — Growing Simplicity (자라는 단순함)
지금은 심플하게 시작한다. 고급 기능(라인별 코드 리뷰, wbox 자동채점, 성적 내보내기)은 점진적으로 추가한다. 기능이 늘어도 기본 UX는 항상 단순하게 유지한다. Progressive Disclosure 원칙 적용.

## 핵심 도메인 단위: 수업 (Class)

- **수업 = 과목 + 분반의 조합** (예: "정보과학 1반", "인공지능 2반")
- 교사 한 명이 여러 수업 담당 가능 (예: 6개 수업)
- 수업이 모든 콘텐츠(차시·수행평가·세션·학생)의 컨테이너
- 관리자가 수업을 생성하고 담당 교사·학생을 배정

---

## 역할별 UX 완전 분리

### 관리자 (Admin)
- 수업 생성·삭제, 담당 교사 배정, 학생 일괄 등록(CSV/XLSX)
- 교사 계정 생성·수정·삭제
- 시스템 설정 (학교명, 동시 시험 제한 등)
- 전체 감사 로그 조회
- **사이드바 있음. 수업 콘텐츠(차시·수행평가)에는 관여 안 함.**

### 일반 교사 (Teacher)
- 홈: 내 수업 카드 목록 (사이드바 없음)
- 수업 진입 후: 차시 생성·편집·공개, 수행평가 생성·배포, 세션 운영·채점, 출석부, 학생 비밀번호 초기화
- 문제 은행 (전역 공유)
- **홈에는 사이드바 없음. 수업 진입 후에만 사이드바 등장.**

### 학생 (Student)
- 홈: 배정된 수업 카드
- 수업 내: 공개된 차시·진행 중인 수행평가만 접근
- 내 제출 결과 확인
- **관리 기능 전혀 없음. 교사가 공개한 것만 보임.**

> **핵심 원칙: "필요한 기능만 보여준다"**
> 역할에 따라 완전히 다른 UI를 제공한다. 같은 컴포넌트도 역할에 따라 편집 버튼 존재 여부가 달라진다.

## 레이아웃 구조

### 홈 화면 (교사·학생 공통)
사이드바 없음. 수업 카드 목록을 그리드로 표시. 상단 바에는 로고, 유저명, 다크모드 토글, 언어 전환만.

### 수업 내부 화면 (교사)
```
┌─────────────┬──────────────────────────────────────────────┐
│  사이드바    │  상단 바: [수업명]  [다크모드] [서버 IP]     │
│  (200px)    ├──────────────────────────────────────────────┤
│             │  콘텐츠 탭: [차시] [수행평가] [세션] [출석부]│
│  📘 차시    ├──────────────────────────────────────────────┤
│  📋 수행평가│                                              │
│  ▶ 세션     │  메인 콘텐츠 영역                            │
│  📅 출석부  │  (선택된 수업 컨텍스트 안)                   │
│  👥 학생    │                                              │
│  ─────────  │                                              │
│  🔙 수업목록│                                              │
└─────────────┴──────────────────────────────────────────────┘
```

### 시험 모드 (학생 응시 중)
사이드바, 상단 네비게이션 모두 숨김. 타이머 + 문제 + 에디터 + 제출 버튼만 표시.

## UX 원칙

### 1. 3-Click Rule
교사가 세션을 시작하기까지 3번 이내의 클릭. `수업 카드 클릭(1) → 세션 탭(2) → 시작 버튼(3)`.

### 2. 빈 상태는 다음 행동 안내
"없습니다" 대신 "아직 차시가 없어요. 첫 차시를 만들어볼까요? [+ 차시 추가]".

### 3. 모든 에러 상태에 복구 안내
원인 + 다음 행동을 함께 표시. "오류가 발생했습니다"만으로는 불충분.

### 4. 시험 중 에러는 특별 처리
제출 실패 시: "제출에 실패했습니다. 작성 내용은 임시저장되었습니다. [다시 시도]". 빨간 고정 배너.

### 5. 로딩·에러 상태 항상 UI에 표시
모든 비동기 작업에 스피너 또는 스켈레톤, 에러 배너 필수.

### 6. 진행 중인 세션 최상단 배너
어느 화면에 있든 세션이 진행 중이면 붉은 고정 배너. "정보과학 1반 — 1차 수행평가 진행 중 · 18/24 제출 · 24분 남음 → 운영 화면"

### 7. 정보 밀도는 화면 유형에 따라 다르게
- 세션 운영·출석부: 고밀도 (테이블/그리드, 학생 30명을 한눈에)
- 학생 응시 화면: 저밀도 (문제 + 에디터에만 집중)
- 홈 카드 화면: 중간 밀도

### 8. 색 + 텍스트 동시 표기 (색각 이상 대응)
상태 표시 시 배지 색과 텍스트를 항상 함께 사용.
- `🟢 Correct` `🔴 Wrong` `🟡 Pending` `⚪ Error`
- 색만으로 상태를 전달하는 UI 요소 금지.

## 시험 모드 격리 (Exam Mode Isolation)

학생이 수행평가 세션에 응시 중일 때:
- 사이드바·헤더 네비게이션 완전 숨김
- 타이머 상단 고정
- 다른 수업/화면으로의 라우팅 차단
- 창 닫기 전 확인 다이얼로그 (Tauri: `onCloseRequested`, 웹: `beforeunload`)
- 제출 버튼 클릭 시 확인 다이얼로그 필수

**동시 시험 제한 (메모 — 추후 DB 설계 시 확정):**
한 학생이 두 세션을 동시에 응시하는 것을 경고하거나 차단. 새 세션 생성 시 해당 학생이 포함된 다른 `RUNNING` 세션 존재 여부를 체크. 관리자 설정에서 `concurrent_session_block: true` (기본값 ON).

## 임시저장 (Draft Auto-save)

모든 문항 유형(단답형·서술형·코드형·빈칸채우기)의 응시 내용은 이중 레이어로 임시저장한다.

**레이어 1 — localStorage (학생 PC, 즉각 저장)**
키: `draft:{sessionId}:{problemId}` — 입력 시마다 저장. 서버 응답 불필요.

**레이어 2 — DB (서버, 30초 주기 동기화)**
`submission_drafts` 테이블에 UPSERT. 다른 PC 접속 또는 localStorage 손실 시 복구용.

**복구 우선순위:** localStorage → 서버 draft → 빈 에디터

**content 필드는 문항 유형에 따라 다형성으로 사용:**
- 단답형: 텍스트 문자열
- 코드형: 코드 문자열 (language 필드 병용)
- 빈칸채우기: JSON 배열 `["답1", "", "답3"]`
- 과제형(파일): 임시저장 불필요 (업로드 자체가 제출)

## 파일 임포트 원칙 (CSV / XLSX)

관리자가 학생 명단을 임포트할 때:

1. **열 인덱스 사용 절대 금지.** 반드시 열 이름으로 매핑.
2. **매핑 사전 운용.** 동의어 열 이름을 동일하게 처리:
   - `student_number`: `["번호", "학번", "No", "number", "student_no"]`
   - `name`: `["이름", "성명", "학생명", "name", "student_name"]`
   - `grade`: `["학년", "grade", "year"]`
   - `class_no`: `["반", "학반", "class", "division"]`
3. **3단계 흐름:** 파일 선택 → 미리보기 + 오류 확인 → 확정 업로드
4. **CSV 인코딩 자동 감지:** UTF-8, EUC-KR(CP949) 자동 변환
5. **XLSX 지원:** NEIS 표준 양식 및 자체 양식 모두 허용 (열 이름 매핑으로 통일)

## 연결 상태 인디케이터

- 학생 → 서버: 10초마다 heartbeat POST
- 연속 3회 실패 시 학생 화면 상단에 배너:
  - 시험 중: 빨간 배너 + "임시저장됨" 안내 + 자동 재시도
  - 일반 수업: 노란 배너
- 교사 출석부 탭: 5초마다 폴링, `last_seen_at` 기준 30초 초과 시 회색 표시
- 교사 상단 바: 서버 연결 상태 인디케이터 (● 연결됨 / ● 끊김)

## 출석부 기능

수업 내 사이드바에 `출석부` 탭 추가.

- **실시간 로그인 현황:** 초록(접속 중) / 회색(미접속) 으로 학생 명단 표시
- **출석 저장:** 교사가 버튼 클릭 시 현재 접속 학생 목록 + 타임스탬프를 DB에 스냅샷 저장
- **학생 명단 관리:** 일반 교사도 학생 명단 수정·추가 가능 (비밀번호 초기화 포함)
- DB 테이블: `attendance_records` (snapshot 방식)

## 서버 IP 표시

- 교사 상단 바에 현재 서버 주소 표시: `http://192.168.1.x:8080`
- 클릭 시 크게 확대 (QR코드 + URL) — 학생들이 접속 주소를 보고 따라 입력하도록
- Rust에서 LAN IP 조회 (`local_ip_address` 크레이트)

## 이중 접속 모드 (Tauri + 웹 브라우저)

- Tauri 앱은 Axum HTTP 서버의 WebView 래퍼
- 교사·관리자는 브라우저에서 `http://[서버IP]:8080` 으로 직접 접속 가능
- 서버 PC와 교사 PC가 다른 경우도 지원
- 시스템 트레이: 열기 / 종료 / Windows 시작 시 자동실행

## 오프라인 우선 (Offline-First)

인터넷 없이 완전 동작해야 한다. 외부 CDN 의존 완전 제거.

- **폰트:** `pretendard` npm 패키지로 번들에 포함
- **코드 폰트:** `JetBrains Mono` 또는 `Fira Code` npm 패키지
- **아이콘:** `@tabler/icons-vue` (현재 사용 중) — npm 패키지이므로 유지
- CDN 링크(`fonts.googleapis.com` 등) 절대 사용 금지

## 다크모드

처음부터 CSS 변수를 라이트/다크 이중 정의한다.

```css
:root {
  --color-surface: #ffffff;
  --color-bg: #f8fafc;
  --color-sidebar: #0f172a;
  --color-text-primary: #111827;
  --color-text-muted: #6b7280;
  --color-border: #e5e7eb;
  --color-accent: #2563eb;
}

[data-theme="dark"] {
  --color-surface: #1e293b;
  --color-bg: #0f172a;
  --color-sidebar: #020617;
  --color-text-primary: #f1f5f9;
  --color-text-muted: #94a3b8;
  --color-border: #334155;
  --color-accent: #3b82f6;
}
```

- `html` 태그의 `data-theme` 속성으로 전환
- `prefers-color-scheme` 으로 시스템 기본값 감지
- 모든 컴포넌트는 토큰만 사용, 하드코딩 색상 완전 금지

## 다국어 지원 (i18n)

- `vue-i18n` 사용
- 모든 UI 텍스트는 `$t('key')` 로만 출력. 한국어 하드코딩 금지.
- 날짜·숫자는 `$d()` · `$n()` 으로 로케일 포맷
- 초기 지원: 한국어(기본) · 영어
- 로케일 파일: `src/locales/ko.json`, `src/locales/en.json`
- 추가 언어는 JSON 파일 하나 추가로 확장

## wbox 채점 결과 UX (추후 구현)

백엔드가 JSON으로 테스트 케이스별 결과를 반환:

```json
{
  "results": [
    { "case": 1, "status": "pass", "input": "5", "expected": "10", "actual": "10" },
    { "case": 2, "status": "fail", "input": "0", "expected": "0", "actual": "1" }
  ]
}
```

프론트엔드는 테이블로 표시. 학생이 어느 케이스에서 실패했는지 확인 후 코드 수정 가능. **UX 전체 확정 후 개발.**

---

## 성적 내보내기 (추후 협의)

두 가지 형식 지원 예정:
1. **배점 기반:** 학생별 총점·문항별 점수 (NEIS 입력용)
2. **정오 매트릭스:** 행=학생, 열=문항, 값=O/X (교사 분석용)

형식 및 열 구조는 채점 기능 개발 시 확정.

## Google Classroom 대비 차별점

| 항목 | Google Classroom | CodeClan LMS |
|---|---|---|
| 코드 제출 | 파일 첨부/링크 | 웹 내장 에디터에서 바로 작성·제출 |
| 코드 채점 | 교사가 수동 실행 | wbox 자동 채점 + 테스트 케이스 결과 표시 |
